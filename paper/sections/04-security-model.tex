\section{Security Model}
\label{sec:security-model}

This section fixes the security model used throughout the paper. The scheme
syntax and the authenticated-encryption and committing-security definitions of
\Cref{sec:prelims} are instantiated here for \TW{} in the lifted
public-permutation model (\Cref{sec:games-pp}), and we define the intermediate
random-oracle world through which every bound is first proved
(\Cref{sec:ro-model}). \Cref{sec:resources} fixes the resource accounting that
all later bounds reference, and \Cref{sec:mu-setting} instantiates the
multi-user indistinguishability game for \TW{}.

\subsection{Public-Permutation Games}
\label{sec:games-pp}

The lifted AE-style public-permutation advantages below compare a real
execution, with sampled $\pi \sample \Perm(\bits^{1600})$ and direct
primitive interface $(\pi,\pi^{-1})$, to a simulator-ideal execution. In
the simulator-ideal execution, the construction interface is the ideal
interface specified by the game and the public primitive interface is
$(\Sim^{\ROflat},\Sim^{-1,\ROflat})$. The simulator
$\Sim^{\ROflat}$ together with its inverse $\Sim^{-1,\ROflat}$ is the
public-primitive simulator fixed in \Cref{app:flat-sponge-lift}; both
directions share an internal lazy table for the random oracle
$\ROflat$ of \Cref{sec:ro-model}. INT-CTXT and the
committing-security games remain one-world real public-permutation
success probabilities; the simulator appears for them only inside the
reductions of \Cref{app:flat-sponge-lift}.

In the real authenticated-encryption games (privacy, INT-CTXT,
all-in-one AE), the challenger additionally samples a key $K \sample
\bits^k$ and exposes the construction oracles $\Enc_K[\pi]$ (the \TW{}
encryption algorithm of \Cref{alg:enc} with the sampled $\pi$) and
$\Ver_K[\pi]$, defined by $\Ver_K[\pi](N, A, C, T) = 1$ if
$\Dec_K[\pi](N, A, C \concat T) \neq \bot$ and $0$ otherwise. The
committing-security games have no challenger-sampled key.

\paragraph{Validity convention.}
All construction-interface inputs are checked against \TW{}'s public
syntax and domain before any construction computation or
construction-internal primitive lookup is performed. An encryption input
is valid when $N \in \bits^\nu$, $A$ is a byte string with
$\|A\| \leq A_{\max}$, and $M$ is a byte string with
$\|M\| \leq M_{\max}$. A verification or decryption input is valid when
$N \in \bits^\nu$, $A$ is a byte string with $\|A\| \leq A_{\max}$, and
the full ciphertext parses as $C \concat T$ with $\|C\| \leq M_{\max}$
and $T \in \bits^{\tauroot}$. In the committing games, each submitted
tuple must be a valid encryption input including
$K \in \bits^k$, and a submitted CMTDs ciphertext must be a valid
verification/decryption ciphertext; in the tag-committing
game below, each submitted body joined to the common tag must be a
valid verification/decryption ciphertext. Invalid encryption queries
are rejected and not recorded, invalid verification queries return $0$,
and invalid committing-game outputs make the game return $0$. Such rejected
inputs contribute no construction cost and create no construction
transcripts. Throughout, an \emph{accepted} encryption query is one that
passes this validity check and any game-specific nonce or user check.

\paragraph{Privacy.}
The privacy game compares the real encryption interface with the fresh
output oracle $\Rand$ that, on every accepted query $(N, A, M)$,
returns a uniformly random byte string of length $\|M\| +
\tauroot/8$. The adversary is \emph{nonce-respecting}:
no two encryption queries may share a nonce, and a query that repeats
an earlier nonce is rejected and not recorded.
\[
  \Advpriv_\TW(\mathcal{A}) =
  \bigl|
    \Pr[\mathcal{A}^{\Enc_K[\pi],\, \pi,\, \pi^{-1}} \Rightarrow 1]
    - \Pr[\mathcal{A}^{\Rand,\, \Sim^{\ROflat},\, \Sim^{-1,\ROflat}} \Rightarrow 1]
  \bigr|.
\]

\paragraph{Ciphertext integrity (INT-CTXT).}
The INT-CTXT game exposes encryption, verification, and the public
permutation interfaces $\pi,\pi^{-1}$.
Each encryption query records the resulting tuple $(N, A, C, T)$. The
adversary wins if some verification query returns $1$ on a tuple $(N,
A, C, T)$ that was not previously recorded by the encryption oracle.
\[
  \Advintctxt_\TW(\mathcal{A}) =
  \Pr[\mathcal{A}^{\Enc_K[\pi],\, \Ver_K[\pi],\, \pi,\, \pi^{-1}} \text{ wins}].
\]
No nonce uniqueness is imposed.

\paragraph{All-in-one AE.}
The all-in-one authenticated-encryption game~\cite{RS06} replaces the
construction interface by the ideal pair $(\Rand, \Replay)$. On each
accepted encryption query $(N,A,M)$, $\Rand$ samples a uniformly
random byte string of length $\|M\| + \tauroot/8$, parses it as
$C \concat T$ with $T \in \bits^{\tauroot}$, records $(N,A,C,T)$, and
returns $C \concat T$. The oracle $\Replay$ returns $1$ exactly on
recorded tuples $(N,A,C,T)$ and $0$ otherwise. The adversary is
nonce-respecting on encryption queries.
\[
  \Advae_\TW(\mathcal{A}) =
  \bigl|
    \Pr[\mathcal{A}^{\Enc_K[\pi],\, \Ver_K[\pi],\, \pi,\, \pi^{-1}} \Rightarrow 1]
    - \Pr[\mathcal{A}^{\Rand,\, \Replay,\, \Sim^{\ROflat},\, \Sim^{-1,\ROflat}} \Rightarrow 1]
  \bigr|.
\]
This lifted all-in-one AE advantage is the headline target of
\Cref{sec:headline}: a Bellare--Namprempre composition
(\Cref{lem:bn-composition}) applied in the random-oracle model to
derived privacy and INT-CTXT adversaries, combined with the flat-sponge
lift, bounds it via the privacy and INT-CTXT envelopes of
\Cref{sec:thms-pp}; the precise bound is \Cref{thm:ae}.

\paragraph{Committing security.}
The $s$-way committing-security game follows \cite[Figs.~4 and~5]{BH22},
extended to $s \geq 2$ tuples, and is stated in the tag-committing form of
\Cref{sec:commit-defs}, which subsumes the
full-ciphertext notion. There is no challenger-sampled key. After the
primitive sampling, the adversary outputs a common tag $T$ together with,
for each claimed opening, a ciphertext body $C_i$ and a tuple
$(K_i, N_i, A_i, M_i)$. The challenger applies the following deterministic
checks, with the eager length rejection justified by \Cref{alg:dec}:
$\Dec_K(N,A,C_i \concat T)$ returns either $\bot$ or a message of length
$\|C_i\|$.
\begin{align*}
  &\pi \sample \Perm(\bits^{1600}), \\
  &\bigl(T,\,(C_i,K_i,N_i,A_i,M_i)_{i=1}^s\bigr)
     \sample \mathcal{A}^{\pi,\, \pi^{-1}}, \\
  &\text{return $0$ if any $C_i \concat T$ is not a valid \TW{} ciphertext}, \\
  &\text{return $0$ if any $(K_i, N_i, A_i, M_i)$ is not in the \TW{} syntactic domain}, \\
  &\text{return $0$ if any $\|M_i\| \neq \|C_i\|$}, \\
  &\text{return $0$ if $(K_i, N_i, A_i, M_i)_{i=1}^s$ are not pairwise distinct}, \\
  &\text{return } \textstyle\bigwedge_{i=1}^s
     \bigl[\Dec_{K_i}[\pi](N_i, A_i, C_i \concat T) = M_i\bigr].
\end{align*}
The advantage is
\[
  \Advtagcmtds{4}_\TW(\mathcal{A})
  = \Pr[\text{the game returns } 1],
\]
over the uniform choice of $\pi$ and $\mathcal{A}$'s coins; no nonce
uniqueness is imposed. The ordinary full-input CMTDs-4 game is the special
case in which all bodies coincide, $C_1 = \cdots = C_s = C$, so the
openings share the entire ciphertext $C \concat T$ rather than only its
tag; its advantage is $\Advcmtds{4}_\TW(\mathcal{A})$.
For $\ell \in \{1, 3\}$, the full-input CMTDs-$\ell$ game is this special
case with the pairwise-distinctness check applied to the
$\WiC_\ell$-images instead---the keys $K_i$ for $\ell = 1$ and the triples
$(K_i, N_i, A_i)$ for $\ell = 3$---keeping the same check order and eager
length rejection; $\Advcmtds{\ell}_\TW(\mathcal{A})$ is defined
analogously.
The CMTs-$\ell$ source games are instantiated
analogously from \Cref{sec:commit-defs}: after the primitive sampling,
the adversary outputs $s$ tuples, the challenger checks syntax and
pairwise $\WiC_\ell$-distinctness, and then returns $1$ iff the
$s$ deterministic encryptions $\Enc_{K_i}[\pi](N_i,A_i,M_i)$ are all
equal.
\Cref{sec:cmtds3} proves the CMTDs-4 bound (\Cref{thm:cmtds4-ro}) and
the T-CMTDs-4 bound (\Cref{thm:tag-cmtds4-ro}) directly,
while \Cref{app:committing-transfer} forwards the remaining \cite{BH22}
committing notions---CMTDs-$\ell$ and the CMTs-$\ell$ source games---to
the same root-collision bound.

\paragraph{Context discoverability.}
The context-discoverability game is the preimage analogue of the
committing game, following the query-independent context selector of
\cite{MLGR23}. It is parameterized by a target tag $T^* \in
\bits^{\tauroot}$ fixed before the primitive sampling, hence
independent of $\pi$ and of the adversary's queries. There is no
challenger-sampled key. After the primitive sampling, the adversary
outputs a ciphertext body $C$ together with a single decryption context
$(K, N, A)$, and the challenger runs the one deterministic decryption
check on the assembled ciphertext $C \concat T^*$.
\begin{align*}
  &\pi \sample \Perm(\bits^{1600}), \\
  &\bigl(C,\, (K, N, A)\bigr) \sample \mathcal{A}^{\pi,\, \pi^{-1}}(T^*), \\
  &\text{return $0$ if $C \concat T^*$ is not a valid \TW{} ciphertext}, \\
  &\text{return $0$ if $(K, N, A)$ is not in the \TW{} syntactic domain}, \\
  &\text{return } \bigl[\Dec_{K}[\pi](N, A, C \concat T^*) \neq \bot\bigr].
\end{align*}
The advantage is the worst case over query-independent targets,
\[
  \Advcdy_\TW(\mathcal{A})
  = \max_{T^* \in \bits^{\tauroot}} \Pr[\text{the game returns } 1],
\]
with each probability taken over the uniform choice of $\pi$ and
$\mathcal{A}$'s coins. By \Cref{alg:dec}, the single check accepts
exactly when the recomputed final root tag of $(K, N, A, C)$ equals
$T^*$, so the game asks the adversary to discover a context whose
ciphertext body extends to the fixed tag $T^*$; the win condition makes
no reference to a claimed message, so the committing game's eager
length and pairwise-distinctness checks are absent. Fixing only the tag
while leaving $(K, N, A)$ and $C$ free makes this the maximal-freedom
discovery target: the restricted notions of \cite{MLGR23}, which
additionally pin two of the three context components, are special cases, so the
headline bound (\Cref{thm:cdy}) dominates each of them.
\Cref{app:flat-sponge-lift} lifts the CDY bound proved in
\Cref{sec:cmtds3} (\Cref{thm:cdy-ro}) to the public-permutation
statement of \Cref{sec:headline}.

\subsection{The TW128 Random-Oracle Model}
\label{sec:ro-model}

The proofs of \Cref{sec:privacy,sec:authenticity,sec:cmtds3} and
\Cref{app:committing-transfer}, together with the headline reductions of
\Cref{sec:headline}, are carried out through an intermediate
random-oracle model whose construction interfaces use the \cite{BDPV11}
bridge of \Cref{sec:sponge} to address a single random oracle. The
flat-sponge lift of \Cref{app:flat-sponge-lift} then converts the
resulting bounds to the public-permutation form.

The model has a single random oracle $\ROflat : \bits^* \to
\bits^\infty$ over the unpadded $H$-input domain of the \cite{BDPV08}
simple-padded sponge interface, with lazy sampling: each distinct
input receives an independent infinite bit string, and repeated queries
return the same string. A direct adversary query is finite-output: on
input $(Y, \ell)$ with finite $\ell$, the oracle returns the first
$\ell$ bits of the lazy-sampled value for $Y$.

For a well-formed \TW{} duplex transcript prefix $X$ in the language of
\Cref{def:wf-transcripts}, at one of the construction transcript lookup
points formalized by \Cref{lem:bridge-decode,tab:keyed-twout}, let
$\flatmap(X) = \bridge[1344, 1344](\mathsf{raw\_flat}(X))$ be the
bridged image of the native flattened input under the
\cite[Theorem~3]{BDPV11} preprocessing map of \Cref{sec:sponge}, and define
\[
  \RF(X) = \ROflat(\flatmap(X)).
\]
The construction interfaces $\Enc_K^{\RO}$, $\Dec_K^{\RO}$, and
$\Ver_K^{\RO}$ are the \TW{} algorithms of \Cref{sec:tw128} with every
root or leaf duplex output replaced by the corresponding requested
projection of $\RF(X)$; the projection length is the duplexing call's
output parameter $\ell$, which may be zero. The bridge and the
typed-restriction convention are formalized in
\Cref{lem:bridge-decode}; the direct-query key-hit predicate ignores
the adversary's requested output length and is formalized as
$\KeyedTWOut$ in \Cref{lem:bridge-decode}.

The random-oracle-model games are the construction-interface variants
of \Cref{sec:games-pp,sec:mu-setting}: the primitive interface is
replaced by direct adversary access to $\ROflat$, and each construction
oracle uses the corresponding ${}^\RO$ algorithm, the ideal oracles
$(\Rand,\Replay)$ being unchanged. Under this substitution
$\Adv^{\mathsf{priv}}_{\RO}$, $\Adv^{\mathsf{int\text{-}ctxt}}_{\RO}$,
and $\Adv^{\mathsf{ae}}_{\RO}$ are the \Cref{sec:games-pp} advantages
with no simulator on the ideal side; the keyless
$\Advcmtds{4}_{\RO}(\mathcal{B})$,
$\Advtagcmtds{4}_{\RO}(\mathcal{B})$, and, for $\ell \in \{1,3,4\}$,
$\Advcmts{\ell}_{\RO}(\mathcal{B})$ are the probabilities that the
corresponding random-oracle committing games return $1$, the CMTs
challenger computing its deterministic encryption checks with
$\Enc^\RO$. For mu-ind, $\Adv^{\mathsf{mu\text{-}ind}}_{\RO}(\mathcal{B})$
denotes the \Cref{sec:mu-setting} game with the same $\mathsf{New}$
interface and per-user nonce discipline---user $d$ using
$(\Enc_{K_d}^{\RO}, \Ver_{K_d}^{\RO})$ when $b = 1$ and
$(\Rand,\Replay)$ when $b = 0$---equivalently, in the two-game
notation of \Cref{sec:mu-ind},
\[
  \Adv^{\mathsf{mu\text{-}ind}}_{\RO}(\mathcal{B}) =
  \bigl|
    \Pr[\mathcal{B}^{\mathsf{Real}^{\mathsf{mu}}_{\RO},\, \ROflat} \Rightarrow 1]
    - \Pr[\mathcal{B}^{\mathsf{Ideal}^{\mathsf{mu}},\, \ROflat} \Rightarrow 1]
  \bigr|,
\]
where the superscripts name the entire multi-user construction
interface in the corresponding world.
Adversary queries to $\ROflat$ are finite-output and counted in
$\qRO(\mathcal{B})$ of \Cref{sec:resources}. Construction-internal
$\RF(\cdot)$ lookups --- those made by $\Enc_K^\RO$, $\Dec_K^\RO$,
$\Ver_K^\RO$, or a committing challenger --- are not.

\subsection{Resources}
\label{sec:resources}

Adversary resources are measured by fixed, execution-uniform caps on
the number and size of oracle queries. An execution-uniform cap is one
that holds on every execution path, including over adversary
randomness, oracle randomness, and adaptive query choices.
Proofs may also introduce local execution-uniform caps, such as
$q_e$ for the number of accepted encryption queries, solely to index a
finite hybrid sequence. Such local caps are part of the adversary's
fixed query bounds, but they are not listed as envelope parameters
unless they affect the final bound.

\begin{description}
\item[$N$:] the construction-interface cost, measured as the number of
  \Keccakp{} calls that the adversary's construction-interface
  queries induce in the real \TW{} computation after the validity
  convention above has been applied. In ideal games, the same accounting
  applies to the corresponding real \TW{} computation on the valid
  query, even when an ideal oracle answers by table lookup or without
  running the construction. $N$ counts at
  per-block granularity, summing each root initialization,
  associated-data block, root message block, leaf initialization, leaf
  message block, and aggregation block across the entire query
  sequence. Valid INT-CTXT verification computations are counted whether
  they accept or reject. In the committing and discoverability games
  there is no adversary
  construction oracle: for CMTDs-$\ell$, $N$ counts only
  the deterministic decryption checks actually performed by the
  challenger on the common ciphertext; for T-CMTDs-4, $N$ counts only
  the deterministic decryption checks actually performed by the
  challenger on the submitted bodies and common tag; for the
  context-discoverability game, $N$ counts only the single deterministic
  decryption check the challenger performs on $C \concat T^*$; while for the
  CMTs-$\ell$ source games of
  \Cref{app:committing-transfer}, $N$ counts only the deterministic
  encryption checks actually used by the challenger to compare the $s$
  ciphertexts.
\item[$\Nprim$:] the number of direct adversary queries to the
  primitive interface (the pair $(\pi, \pi^{-1})$ in
  public-permutation games; the simulator's $\ROflat$ table is internal
  and is not exposed to the application adversary in those games).
\item[$\Ntot = N + \Nprim$:] the total query-sequence cost. The
  flat-sponge lift of \Cref{app:flat-sponge-lift} is applied at
  $\Ntot$ because the indifferentiability replacement sees both the
  construction-internal sponge calls counted by $N$ and the adversary's
  direct primitive-interface queries counted by $\Nprim$ in one
  chronological public-primitive transcript.
\item[$\qv(\mathcal{A})$, $\Qv$:] the number of verification queries
  with valid inputs, and a cap on it. Used in INT-CTXT and absorbed into
  the AE composition of \Cref{sec:headline}. Every valid verification
  query contributes at least one verification/decryption
  construction call in the $N$ accounting, so $\qv(\mathcal{A}) \leq N$.
\item[$\qRO(\mathcal{B})$, $\QRO$:] the number of direct
  finite-output queries to $\ROflat$ by a random-oracle-model adversary
  $\mathcal{B}$, and a cap on it. Repeated direct queries on the same
  $Y$ are counted again. Construction-internal $\RF(\cdot)$ lookups are
  not counted. After the derived-adversary construction, $\qRO$ counts only
  simulator-induced $\ROflat$ calls from the adversary's direct
  primitive-interface queries, not the construction calls already
  counted in $N$ for the flat-sponge lift.
\item[$\nleaf(\mathcal{B})$, $\Nleaf$:] the number of distinct
  construction-generated final leaf transcripts in the execution of
  $\mathcal{B}$, and a cap on it. Distinct final leaf transcripts are
  generated by valid encryption computations, by valid INT-CTXT
  verification computations (whether they accept or reject), and by
  committing, tag-committing, and discoverability challenger
  construction checks that pass the early validity checks.
\end{description}

For a byte string $X$,
\[
  \leaves(X) = \max\bigl(0,\,
    \lceil \|X\| / \beta \rceil - 1\bigr)
\]
counts the leaf chunks induced by a body byte string of length $\|X\|$.
Thus $\nleaf$ is bounded by the sum of $\leaves(X)$ over the valid
construction computations that create final leaf transcripts, where
$X$ is the plaintext body for encryption computations and the submitted
ciphertext body for verification and committing or discoverability
decryption checks.
Each distinct final leaf transcript contains a final leaf duplex call
counted in $N$, so $\nleaf(\mathcal{B}) \leq N$ unconditionally. The
ratio $\nleaf/N$ is largest when most processed bytes lie in leaf
chunks, where each full leaf contributes $1 + \beta/\rho = 50$ duplex
calls to $N$ per final leaf transcript.

All theorems are stated for adversaries whose execution-uniform caps
hold at the stated bounds.

\subsection{Multi-User Setting}
\label{sec:mu-setting}

This subsection instantiates the mu-ind game of \cite{BT16}
(\Cref{sec:mu-ind}) for \TW{} in the lifted public-permutation model,
using the same ideal-side simulator convention as the single-user
AE-style games of \Cref{sec:games-pp}. The challenger samples a
challenge bit $b \sample \bits$. If $b = 1$, it samples a permutation
$\pi \sample \Perm(\bits^{1600})$ and exposes $(\pi,\pi^{-1})$ as the
primitive interface; if $b = 0$, it exposes
$(\Sim^{\ROflat},\Sim^{-1,\ROflat})$ as the primitive interface. The
adversary $\mathcal{A}$ creates users on demand via $\mathsf{New}$, and
the game maintains an active-user counter $v$. We write $D$ for an
execution-uniform cap on the number of $\mathsf{New}$ calls. Equivalently,
the game may sample independent keys $K_1,\dots,K_D$ at the start of the
experiment and let the $d$-th $\mathsf{New}$ call activate $K_d$; inactive
indices are never valid users.

The mu-ind construction oracles inherit the validity convention of
\Cref{sec:games-pp}. The encryption oracle on input $(d, N, M, A)$
rejects, without recording, if $(N,A,M)$ is outside the \TW{} encryption
domain, if $d \notin \{1, \dots, v\}$, or if $(d, N) \in V$; otherwise
it sets $C = \Enc_{K_d}[\pi](N, A, M)$ when $b = 1$, and samples
$C \sample \bits^{8\|M\|+\tauroot}$ when $b = 0$. The game records
$(d, N)$ in $V$ and $(d, N, C, A)$ in $W$, and returns $C$. The
verification oracle on input $(d, N, C, A)$ rejects if $d \notin
\{1, \dots, v\}$ or if $(N,A,C)$ is not a valid \TW{}
verification/decryption input; if $(d, N, C, A) \in W$ it returns
$\mathsf{true}$; if $b = 0$ it returns $\mathsf{false}$; otherwise it
returns $[\Dec_{K_d}[\pi](N, A, C) \neq \bot]$. Nonce uniqueness is
enforced per user by the set $V$.

After querying these oracles, $\mathcal{A}$ outputs $b' \in \bits$.
The mu-ind advantage of $\mathcal{A}$ against \TW{} is
\[
  \Advmuind_\TW(\mathcal{A})
  = |2 \Pr[b' = b] - 1|.
\]

Resource caps split per potential user $d \leq D$: $N^{(d)}$,
$\qv^{(d)}$, $\nleaf^{(d)}$ are the per-user counts under
\Cref{sec:resources}, with inactive users contributing zero. The global
totals $N = \sum_{d=1}^D N^{(d)}$, $\qv = \sum_{d=1}^D \qv^{(d)}$,
$\nleaf = \sum_{d=1}^D \nleaf^{(d)}$ appear in the final bound: $N$
through the lift term $\Lift_c(\Ntot)$, and $\nleaf$ and $\qv$ through
the leaf-collision and verification-tag terms. $\Nprim$
remains a single shared count over the primitive interface.

The mu-ind treatment applies only to indistinguishability. The
committing-security games already give the adversary control of all
$s$ keys, so the $s$-way CMTDs-$\ell$ statements of
\Cref{sec:games-pp} are themselves the multi-key committing statement
and need no separate $\mathsf{mu\text{-}cmt}$ notion.
